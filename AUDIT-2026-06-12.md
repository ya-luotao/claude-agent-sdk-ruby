## 一、高严重度 Bug(9 个)

### H1. 流式输入 task 永不取消 → `query()` 永久挂起且异常被吞
`lib/claude_agent_sdk.rb:360-362`、`lib/claude_agent_sdk/query.rb:1027-1035`

Enumerator prompt 时,`query()` 用裸 `Async { query_handler.stream_input(prompt) }` 启动输入 task 且不保留句柄;`Query#close` 只 stop `@task`(读 task)。async gem 的根 task 要等**所有**子 task 结束才返回,所以当读循环抛异常(CLI crash)而用户的 enumerator 正阻塞(如 `Queue#pop`、阻塞 IO)时,`query()` 永久挂起,`CLIConnectionError` 退化成一条 async console warning。**验证者用 live repro 复现:模拟 CLI crash 后 6 秒 `query()` 仍阻塞。** Python 不受影响:`query.spawn_task(...)` + `close()` cancel 所有 `_child_tasks`。
修复:把 stream_input task 注册到 Query 上,在 `close` 中 stop。

### H2. ProcessError 抑制分支:in-flight 控制请求挂 1200 秒 + 多轮会话真实 crash 被吞
`lib/claude_agent_sdk/query.rb:245-257`

`@first_result_received` 为 true 时 rescue ProcessError 只 warn,既不 signal `@pending_control_responses` 也不 enqueue error。后果:(1) CLI 死亡时 in-flight 的 `interrupt`/`set_model` 阻塞整整 `DEFAULT_CONTROL_REQUEST_TIMEOUT_SECONDS = 1200s`(20 分钟)后抛误导性的 ControlRequestTimeoutError;(2) `@first_result_received` 整个 session 粘滞,多轮 Client 会话中第 N>1 轮的真实 crash(segfault/OOM)只产生一条 warning,消费者看到 'end' 哨兵以为正常结束。该分支源自 commit 00ae0a5(吸收 StructuredOutput 后 CLI 故意 exit 1),但实现过宽:Python 的等价抑制按 `_last_error_result_text` 限定作用域(仅紧跟 is_error result 的退出被软化,新消息到达即重置),且任何读循环异常都**无条件** signal 所有 pending control events 并投递 `{type:'error'}`。
修复:无条件 signal pending conditions;把抑制范围从"整个 session"收窄到"紧跟 error result 的退出"。

### H3. 60 秒 stream-close 超时在 mid-turn 关闭 stdin,杀死 hooks/SDK MCP 控制协议
`lib/claude_agent_sdk/query.rb:977-990`(及 29-30 默认值)

`wait_for_result_and_end_input` 用 `with_timeout(60s)` 等首个 result,超时后 ensure 仍调用 `@transport.end_input` 关闭 stdin。但 hook 回复、can_use_tool 回复、SDK MCP tool 结果都要写 stdin——首轮超过 60 秒(agentic 任务很常见)后所有控制回复抛 `CLIConnectionError('ProcessTransport is not ready for writing')`,hooks 和进程内 MCP tools 静默失效。Python 在 commit c3d96cb 修复过完全相同的问题。
修复:配置了 hooks/can_use_tool/SDK MCP servers 时不要在超时后关 stdin(对齐 Python 行为)。

### H4. `Observer#on_error` 文档化、已实现,但任何代码路径都不会调用它
`lib/claude_agent_sdk.rb:368-392`、`observer.rb:31-33`、`instrumentation/otel.rb:84-89`

`notify_observers` 只有 `:on_user_prompt`/`:on_message`/`:on_close` 三种调用(grep 确认)。CLI 死亡或解析错误时,错误直接穿透 `query()` 的 ensure(只调 on_close)传播出去,observer 永远不知道。后果:crash 的会话在 OTel 里 root span 状态为 UNSET(多数后端显示为 OK);Client 模式下用户出错后不调 disconnect 的话 root span 永不结束。
修复:在 query() 接收循环、Client#receive_messages/receive_response 的 rescue 中调用 `notify_observers(:on_error, e)`。

### H5. OTelObserver 状态跨 query 不重置——后续所有 trace 打上第一个 prompt 的标签
`lib/claude_agent_sdk/instrumentation/otel.rb:57-63, 91-97, 125, 197`

`on_close` 重置了 `@tool_spans`/`@root_span`/`@root_context` 但漏了 `@first_user_input`/`@last_assistant_text`。docs 和 example 都示范"创建一个 OTelObserver 实例放进 options 复用"——第二次 query 时 `on_user_prompt` 因 `@first_user_input` 已设而提前返回,stale prompt 被当作新 trace 的 `input.value`。
修复:`on_close` 里重置全部缓冲状态;文档推荐 factory lambda 形式。

### H6. `append_allowed_tools` 发出的 `--append-allowed-tools` 被 CLI 2.1.175 拒绝
`lib/claude_agent_sdk/command_builder.rb:232-236`

实测:`claude --append-allowed-tools Bash -p hi` → `error: unknown option '--append-allowed-tools'`,设置该选项的用户直接连接失败。该 flag 在 CLI 2.1.175 二进制里 0 次出现;Python SDK 从来没有这个选项——它是 commit 5b084c4("v0.1.18 feature parity")的误植。单元测试只断言了 flag 被拼进数组,没验证 CLI 接受。
修复:删除该选项(或映射到真实存在的 CLI 语义),CHANGELOG 标注 breaking。

### H7. 用户 block 里 `break` → LocalJumpError 而非停止迭代
`lib/claude_agent_sdk.rb:372, 530`

`query()` 和 `Client#receive_messages` 经 `FiberBoundary.invoke` 在另一个 Thread 上执行用户 block,proc 里的 `break` 跨线程抛 `LocalJumpError("break from proc-closure")`。SDK 已知此坑:`Client#receive_response`(542-552)专门重构过把 break 留在 hop 外,`fiber_boundary.rb:27-29` 也记录了该约束——但这两处漏掉了。
修复:与 receive_response 相同的结构调整。

### H8. CLI stdout 按 locale 编码解码,C/POSIX locale 下非 ASCII 输出崩溃读循环
`lib/claude_agent_sdk/subprocess_cli_transport.rb:176, 421-423, 460`

`Open3.popen3` 管道编码取 `Encoding.default_external`,transport 从未覆盖(全文件无 `set_encoding`/`binmode`)。最小 Docker 镜像、systemd 服务、CI runner 常见 LANG=C → US-ASCII,CLI 输出第一个非 ASCII 字节(非英文回复、emoji)就让 `line.strip` 抛 `Encoding::CompatibilityError`——它不是 IOError,460 行的 rescue 接不住。验证者端到端复现。Client 模式下被 query.rb:258 的 rescue StandardError 接住转成 error 消息,但读循环同样死亡。
修复:spawn 后对 stdout/stderr `set_encoding('UTF-8', invalid: :replace, undef: :replace)`(对齐 Python 的 errors='replace')。

### H9. `ToolPermissionContext` 的展示字段永远是 nil
`lib/claude_agent_sdk/query.rb:369-374`(类型定义 `types.rb:561-570`)

类型声明并文档化了 CLI 2.1.110+ 的 `title`/`display_name`/`description`/`blocked_path`/`decision_reason`,但 `handle_permission_request` 构造 context 时只填 signal/suggestions/tool_use_id/agent_id,CLI 发来的数据被丢弃。Python(_internal/query.py:390-405)全部填充。关联:`suggestions` 也以 raw camelCase wire hash 传给回调,而非 `PermissionUpdate` 对象(Python #920)。
修复:从 control request 提取并填充全部字段;把 suggestions 解析为 PermissionUpdate。

## 二、中严重度 Bug(约 20 个,去重)

### 并发 / 控制协议
- **M1** `lib/claude_agent_sdk.rb:349-358` — String prompt + hooks/SDK MCP servers 时,`query()` 内联等待首个 result 才开始投递消息(Python 用 spawned task),消息全部延迟到完成后才可见,且期间 hook 回调无法处理。
- **M2** `query.rb:718-729` — 在 hook/can_use_tool 回调内部调用 client 控制方法(interrupt/set_model)时,请求**已写给 CLI 之后**才抛 opaque RuntimeError,状态不一致。应在写之前检测并给出清晰错误(或支持重入)。
- **M3** `query.rb:707-743` — `send_control_request` 存在 lost-wakeup 竞态:`Async::Condition` 是边沿触发,response 若在 sender 到达 `condition.wait` 之前到达即丢失(当前靠"同 fiber 内写后立刻 wait"的调度巧合掩盖)。应改为电平语义(先查结果槽再 wait)。
- **M4** `lib/claude_agent_sdk.rb:729-739` — `Client#connect` 内联消费 Enumerator prompt:交互式流(等响应再发下一条)直接死锁 connect;且 Hash 消息用 `to_s` 而非 `JSON.generate` 序列化,CLI 收到 `{:type=>"user"}` 这样的 Ruby inspect 格式。
- **M5** `query.rb:993-1004` — `stream_input` 在 reactor fiber 上直接迭代用户 enumerator,绕过 SDK 在其他所有用户回调处提供的 FiberBoundary 保证;用户代码里任何阻塞调用都会饿死整个 reactor。
- **M6** `query.rb:74-87` — `HookMatcher`/per-hook `timeout` 字段未随 initialize 请求转发给 CLI(Python 发送),用户设置的 hook 超时被静默忽略。

### SDK MCP Server
- **M7** `query.rb:789-794` — tool handler 抛异常变成 JSON-RPC `-32603` 协议错误;Python 返回 in-band `isError: true` tool result,模型能看到错误并自我纠正。Ruby 行为让模型只看到协议失败。
- **M8** `sdk_mcp_server.rb:87-89, 120-122, 152-154` — handler 返回 string-keyed hash(`{'content' => ...}`)被拒,而管道其余部分(types.rb、query.rb)明确双向支持 string/symbol key。
- **M9** `sdk_mcp_server.rb:255-339` — 文档化的公共 API `handle_json` 对 resources/prompts 完全坏掉:引用了不存在的 `MCP::ResourceContents` 常量(任何 mcp gem 版本都没有),`resources/read` 必然 NameError;Resource 子类化方式还会破坏 `resources/list`。

### SessionStore / Sessions
- **M10** `sessions.rb:322-326` — `created_at` 只从 transcript 首行提取,首条记录是 metadata-only 时返回 nil。Python #907 扫描整个 head,该修复未 port。
- **M11** `sessions.rb:977-1009` — `find_session_file` 遇到 0 字节 stub 文件即停,Python(和 Ruby 自己的 mutation 路径)会继续找非空文件。
- **M12** `sessions.rb:857, 879, 982` — projects 目录不存在时读 API 抛 `Errno::ENOENT`;Python 返回 `[]`/`nil`。新机器上首次调用 `list_sessions` 即崩。

### Transport / 配置
- **M13** `subprocess_cli_transport.rb:172-176` — `ClaudeAgentOptions.user` 被接受但完全忽略(Python 以该 OS 用户 spawn CLI)。静默不生效比报错更糟。
- **M14** `subprocess_cli_transport.rb:501-519` — `check_claude_version` 无超时、无跳过机制:`claude -v` 挂起则 connect 永久挂起。Python 有超时 + `CLAUDE_AGENT_SDK_SKIP_VERSION_CHECK`。
- **M15** `subprocess_cli_transport.rb:421, 441-448` — `max_buffer_size`(1MB)在 `IO#each_line` 把**整行读进内存之后**才检查,恶意/异常的超长行可造成无界内存分配。需用定长 `readpartial` 分块读。
- **M16** `types.rb:1599-1611` — `Configuration.default_options` 的 Array/嵌套 Hash 值被所有 `ClaudeAgentOptions` 实例**共享引用**,某次会话 `options.allowed_tools << 'Bash'` 会污染全局默认值。需要 deep dup。
- **M17** `command_builder.rb:175-190` — Hash 形式的 `thinking` 配置被静默丢弃(连 `max_thinking_tokens` fallback 也被压制)。
- **M18** `message_parser.rb:159-164` — `advisor_tool_result` content block 被解析成 UnknownBlock;`server_tool_result` 分支是死代码(CLI 实际发的是前者)。

### 文档 / 测试自身的 bug
- **M19** `README.md:48-56` — Client 示例用无终止条件的 `receive_messages`,照抄即永久挂起。应改用 `receive_response`。
- **M20a** `spec/unit/query_spec.rb:538-541` — cancellation spec 的 fixture 回调会立刻 crash 而非 sleep,断言靠竞态通过(test bug)。
- **M20b** `spec/unit/query_spec.rb:272-322` — MCP dispatch 测试 stub 掉 `call_tool` 并断言与真实 server 行为相矛盾的结果(掩盖了 M7/M8)。

## 五、测试缺口(按风险排序)

1. **控制协议 round-trip**(`send_control_request` ↔ `handle_control_response`)零测试——SDK 最核心的机制(high)。
2. **`handle_permission_request`**(can_use_tool wire 转换)完全未测——H9 能存在这么久正是因为它(high)。
3. **CLI mid-stream 死亡**:ProcessError-before-result 须解锁 pending 控制请求 / after-result 须按设计抑制——两个方向都没测(high,与 H2 相关)。
4. **`Query#close` 解锁阻塞中的 `receive_messages` 消费者**——有此 hang 的修复史却无回归测试(high)。
5. transport 读写错误路径:非零退出 ProcessError + stderr、1MB 缓冲溢出、多行 JSON 累积、write guard(medium)。
6. `wait_for_result_and_end_input` 行为(stdin 关闭时机 × hooks/SDK MCP)——H3 的回归防线(medium)。
7. hook 回调抛 StandardError → error control_response 路径(medium)。
8. SDK MCP 路由错误响应(unknown server/method → -32601,handler 异常 → -32603)(low)。
9. close 升级阶梯(graceful → SIGTERM → SIGKILL)与 stderr 回调隔离(low)。
10. 既有 test bug:M20a(竞态通过)、M20b(stub 与现实矛盾)、`subprocess_cli_transport_spec.rb:607-640` 死锁回归测试有 vacuous-pass 风险。
