# Full Codebase Audit Report ‚Äî claude-agent-sdk-ruby v0.17.0

Date: 2026-06-12
Method: 9-dimension parallel review (async/concurrency correctness, subprocess resources, SessionStore, types/parsing, SDK MCP server, public API, Observer/OTel, test gaps, Python parity) + independent adversarial verification of every finding (96 agents total). 87 raw findings, 86 verified (1 empirically refuted), deduplicated to ~55 distinct issues. Key findings carry live repros or were verified against the Python source; `--append-allowed-tools`, the dead `on_error` interface, and the encoding issue were manually re-confirmed in the main loop.

Baseline: 820 examples all green, RuboCop clean.

## 0. Python SDK Latest (0.2.88 ‚Üí 0.2.99)

**Conclusion: nothing to port.**

- 11 releases are almost entirely bundled-CLI bumps (2.1.164 ‚Üí 2.1.175).
- `mcp>=1.23.0,<2.0.0` pin (#1028): guards against the Python MCP SDK 2.0 breaking change. Ruby's pessimistic `mcp ~> 0.4` constraint already caps below 1.0 ‚Äî no action needed. (Note: `~> 0.4` floats up to 0.18.x; a pre-1.0 gem's minors can break compatibility. Core dispatch paths were verified on 0.4.0/0.7.0/0.18.0; consider tightening or adding a floor test in CI.)
- asyncio+trio dual test runs (#1021): pure Python test infrastructure, irrelevant to Ruby.
- `session_store_conformance.py` changed only a docstring.

## 1. High-Severity Bugs (9)

### H1. Streaming-input task is never cancelled ‚Üí `query()` hangs forever and swallows the error
`lib/claude_agent_sdk.rb:360-362`, `lib/claude_agent_sdk/query.rb:1027-1035`

For Enumerator prompts, `query()` spawns the input task via a bare `Async { query_handler.stream_input(prompt) }` without keeping a handle; `Query#close` only stops `@task` (the read task). The async gem's root task waits for **all** children before returning, so when the read loop raises (CLI crash) while the user's enumerator is blocked (e.g. `Queue#pop`, blocking IO), `query()` never returns and the `CLIConnectionError` decays into an async console warning. **The verifier reproduced this live: 6 seconds after a simulated CLI crash, `query()` was still blocked.** Python is immune: `query.spawn_task(...)` + `close()` cancels all `_child_tasks`.
Fix: register the stream_input task on the Query and stop it in `close`.

### H2. ProcessError suppression branch: in-flight control requests hang 1200s + real crashes swallowed in multi-turn sessions
`lib/claude_agent_sdk/query.rb:245-257`

When `@first_result_received` is true, the `rescue ProcessError` only warns ‚Äî it neither signals `@pending_control_responses` nor enqueues an error. Consequences: (1) an in-flight `interrupt`/`set_model` blocks for the full `DEFAULT_CONTROL_REQUEST_TIMEOUT_SECONDS = 1200s` (20 minutes) before raising a misleading ControlRequestTimeoutError when the CLI dies; (2) `@first_result_received` is sticky for the whole session, so a real crash (segfault/OOM) in turn N>1 of a multi-turn Client session produces only a warning, and consumers see the 'end' sentinel as if the session ended cleanly. The branch came from commit 00ae0a5 (absorbing the CLI's intentional exit-1 after a StructuredOutput error result) but overshoots its intent: Python scopes the equivalent softening via `_last_error_result_text` (only the exit immediately following an is_error result is softened; any new message resets the marker), and **unconditionally** signals all pending control events plus delivers `{type:'error'}` on every read-loop exception.
Fix: signal pending conditions unconditionally; narrow the suppression from "whole session" to "exit directly following an error result".

### H3. 60-second stream-close timeout closes stdin mid-turn, killing the hooks/SDK MCP control protocol
`lib/claude_agent_sdk/query.rb:977-990` (defaults at 29-30)

`wait_for_result_and_end_input` waits for the first result under `with_timeout(60s)` and, on timeout, the ensure still calls `@transport.end_input`. But hook replies, can_use_tool replies, and SDK MCP tool results are all written to stdin ‚Äî once a one-shot query's first turn runs past 60 seconds (routine for agentic tasks), every control reply raises `CLIConnectionError('ProcessTransport is not ready for writing')` and hooks/in-process MCP tools silently stop working. Python fixed exactly this in commit c3d96cb.
Fix: never close stdin on timeout when hooks/can_use_tool/SDK MCP servers are configured (align with Python).

### H4. `Observer#on_error` is documented and implemented but never invoked on any code path
`lib/claude_agent_sdk.rb:368-392`, `observer.rb:31-33`, `instrumentation/otel.rb:84-89`

`notify_observers` is only ever called with `:on_user_prompt`/`:on_message`/`:on_close` (grep-confirmed). When the CLI dies or a parse error occurs, the error propagates straight through `query()`'s ensure (which only calls on_close). Consequence: crashed sessions' OTel root spans finish with UNSET status (most backends display as OK); in Client mode, a user who never calls disconnect after an error leaves the root span never finished at all.
Fix: call `notify_observers(:on_error, e)` in query()'s receive loop rescue and in Client#receive_messages/receive_response.

### H5. OTelObserver state is not reset across queries ‚Äî every later trace gets the first query's prompt
`lib/claude_agent_sdk/instrumentation/otel.rb:57-63, 91-97, 125, 197`

`on_close` resets `@tool_spans`/`@root_span`/`@root_context` but misses `@first_user_input`/`@last_assistant_text`. The docs and example both demonstrate "create one OTelObserver instance and reuse it via options" ‚Äî on the second query, `on_user_prompt` returns early because `@first_user_input` is still set, and the stale buffered prompt becomes the new trace's `input.value`.
Fix: reset all buffered state in `on_close`; recommend factory lambdas in docs.

### H6. `append_allowed_tools` emits `--append-allowed-tools`, rejected by CLI 2.1.175
`lib/claude_agent_sdk/command_builder.rb:232-236`

Verified live: `claude --append-allowed-tools Bash -p hi` ‚Üí `error: unknown option '--append-allowed-tools'` ‚Äî any user setting this option gets an instant connection failure. The flag has 0 occurrences in the CLI 2.1.175 binary; the Python SDK never had this option ‚Äî it was a mis-port in commit 5b084c4 ("v0.1.18 feature parity"). The unit test only asserted the flag was pushed into the argv array, never that the CLI accepts it.
Fix: remove the option (breaking change; note in CHANGELOG).

### H7. `break` inside the user block raises LocalJumpError instead of stopping iteration
`lib/claude_agent_sdk.rb:372, 530`

`query()` and `Client#receive_messages` invoke the user block on another thread via `FiberBoundary.invoke`; `break` in a proc across threads raises `LocalJumpError("break from proc-closure")`. The SDK already knows this hazard: `Client#receive_response` (542-552) was specifically restructured to keep break outside the hop, and `fiber_boundary.rb:27-29` documents the constraint ‚Äî but these two call sites were missed.
Fix: same structural change as receive_response.

### H8. CLI stdout decoded with locale-dependent encoding; non-ASCII output crashes the read loop under C/POSIX locale
`lib/claude_agent_sdk/subprocess_cli_transport.rb:176, 421-423, 460`

`Open3.popen3` pipe encoding follows `Encoding.default_external`, which the transport never overrides (no `set_encoding`/`binmode` anywhere in the file). Minimal Docker images, systemd services, and CI runners often have `LANG=C` ‚Üí US-ASCII, where the first non-ASCII byte from the CLI (non-English replies, emoji) makes `line.strip` raise `Encoding::CompatibilityError` ‚Äî which the `rescue IOError` at 460 cannot catch. End-to-end reproduced by the verifier. In Client mode it's caught by query.rb:258's `rescue StandardError` and converted into an error message, but the read loop dies the same way.
Fix: `set_encoding` UTF-8 on stdout/stderr after spawn (matches Python's UTF-8 TextReceiveStream).

### H9. `ToolPermissionContext` display fields are always nil
`lib/claude_agent_sdk/query.rb:369-374` (type at `types.rb:561-570`)

The type declares and documents the CLI 2.1.110+ `title`/`display_name`/`description`/`blocked_path`/`decision_reason` fields, but `handle_permission_request` constructs the context with only signal/suggestions/tool_use_id/agent_id, discarding what the CLI sends. Python (_internal/query.py:390-405) populates all of them. Related: `suggestions` are passed as raw camelCase wire hashes instead of `PermissionUpdate` objects (Python #920).
Fix: populate all fields from the control request; parse suggestions into PermissionUpdate.

## 2. Medium-Severity Bugs (~20, deduplicated)

### Concurrency / control protocol
- **M1** `lib/claude_agent_sdk.rb:349-358` ‚Äî String prompt with hooks/SDK MCP servers: `query()` waits inline for the first result before delivering any messages (Python uses a spawned task); delivery is deferred until completion and hook callbacks cannot be processed during the wait.
- **M2** `query.rb:718-729` ‚Äî calling client control methods (interrupt/set_model) from inside hook/can_use_tool callbacks raises an opaque RuntimeError **after** the request was already written to the CLI, leaving inconsistent state. Detect before writing (or support reentrancy).
- **M3** `query.rb:707-743` ‚Äî lost-wakeup race in `send_control_request`: `Async::Condition` is edge-triggered; a response arriving before the sender reaches `condition.wait` is lost (currently masked by same-fiber scheduling luck). Switch to level-triggered semantics (check the result slot before waiting).
- **M4** `lib/claude_agent_sdk.rb:729-739` ‚Äî `Client#connect` consumes an Enumerator prompt inline: interactive streams (that wait for a response before yielding) deadlock connect; Hash messages are serialized with `to_s` instead of `JSON.generate`, so the CLI receives Ruby inspect format like `{:type=>"user"}`.
- **M5** `query.rb:993-1004` ‚Äî `stream_input` iterates the user enumerator directly on the reactor fiber, bypassing the FiberBoundary guarantee the SDK makes at every other user-callback site; any blocking call in user code starves the whole reactor.
- **M6** `query.rb:74-87` ‚Äî `HookMatcher`/per-hook `timeout` is not forwarded to the CLI in the initialize request (Python sends it); user-set hook timeouts are silently ignored.

### SDK MCP server
- **M7** `query.rb:789-794` ‚Äî tool handler exceptions become JSON-RPC `-32603` protocol errors; Python returns in-band `isError: true` tool results so the model can see the error and self-correct. Ruby's behavior shows the model only a protocol failure.
- **M8** `sdk_mcp_server.rb:87-89, 120-122, 152-154` ‚Äî handlers returning string-keyed hashes (`{'content' => ...}`) are rejected, while the rest of the pipeline (types.rb, query.rb) explicitly supports both key styles.
- **M9** `sdk_mcp_server.rb:255-339` ‚Äî the documented public API `handle_json` is completely broken for resources/prompts: it references the nonexistent constant `MCP::ResourceContents` (no mcp gem version has ever had it), so `resources/read` is a guaranteed NameError; the Resource subclassing approach also breaks `resources/list`.

### SessionStore / sessions
- **M10** `sessions.rb:322-326` ‚Äî `created_at` is extracted only from the first transcript line; nil when the first record is metadata-only. Python #907 scans the whole head; the fix was never ported.
- **M11** `sessions.rb:977-1009` ‚Äî `find_session_file` stops at 0-byte stub files; Python (and Ruby's own mutation path) continues to a non-empty file.
- **M12** `sessions.rb:857, 879, 982` ‚Äî read APIs raise `Errno::ENOENT` when the projects directory does not exist; Python returns `[]`/`nil`. First call to `list_sessions` on a fresh machine crashes.

### Transport / configuration
- **M13** `subprocess_cli_transport.rb:172-176` ‚Äî `ClaudeAgentOptions.user` is accepted but completely ignored (Python spawns the CLI as that OS user). Silent no-op is worse than an error.
- **M14** `subprocess_cli_transport.rb:501-519` ‚Äî `check_claude_version` has no timeout and no skip mechanism: a hung `claude -v` hangs connect forever. Python has a timeout + `CLAUDE_AGENT_SDK_SKIP_VERSION_CHECK`.
- **M15** `subprocess_cli_transport.rb:421, 441-448` ‚Äî `max_buffer_size` (1MB) is checked only **after** `IO#each_line` has read the entire line into memory; malicious/abnormal oversized lines cause unbounded allocation. Needs bounded chunk reads.
- **M16** `types.rb:1599-1611` ‚Äî `Configuration.default_options` Array/nested-Hash values are **shared by reference** across all `ClaudeAgentOptions` instances; `options.allowed_tools << 'Bash'` in one session pollutes the global defaults. Needs a deep dup.
- **M17** `command_builder.rb:175-190` ‚Äî Hash-form `thinking` config is silently dropped (and even suppresses the `max_thinking_tokens` fallback).
- **M18** `message_parser.rb:159-164` ‚Äî `advisor_tool_result` content blocks parse as UnknownBlock; the `server_tool_result` branch is dead code (the CLI actually emits the former).

### Docs / test-suite bugs
- **M19** `README.md:48-56` ‚Äî the Client example uses `receive_messages` with no termination condition; copy-pasting it hangs forever. Should use `receive_response`.
- **M20a** `spec/unit/query_spec.rb:538-541` ‚Äî the cancellation spec's fixture callback crashes instantly instead of sleeping; the assertion passes via a race (test bug).
- **M20b** `spec/unit/query_spec.rb:272-322` ‚Äî the MCP dispatch tests stub `call_tool` and assert results the real server contradicts (masking M7/M8).

## 4. Low-Severity Issues (summary)

**Transport**
- L1 `subprocess_cli_transport.rb:402-411` ‚Äî `end_input` closes/nils `@stdin` without taking `@stdin_mutex`, violating the transport's own locking protocol.
- L2 `subprocess_cli_transport.rb:165` ‚Äî `CLAUDE_CODE_ENTRYPOINT` uses `||=`, so an inherited env value (e.g. 'cli') overrides 'sdk-rb', mis-attributing telemetry.

**Query / control protocol**
- L3 `query.rb:338-349` ‚Äî cancelled control requests write a 'Cancelled' control_response; Python deliberately writes nothing.
- L4 `query.rb:647-650` ‚Äî unknown hook events lose all event-specific fields and arrive with `hook_event_name` nil (should pass the raw payload through).

**SDK MCP**
- L5 `sdk_mcp_server.rb:344-347, 211-214` ‚Äî pre-built JSON Schemas with symbol values (`type: :object`) are silently mangled into garbage schemas.

**OTel**
- L6 `instrumentation/otel.rb:101-126` ‚Äî a second InitMessage without an intervening ResultMessage overwrites an unfinished root span (never exported).
- L7 `instrumentation/otel.rb:245-253` ‚Äî Array tool-result `output.value` is serialized via Ruby `inspect`, inconsistent with the JSON input.

**Sessions**
- L8 `sessions.rb:981-1008` ‚Äî `find_session_file` with an explicit directory silently falls back to scanning all project dirs (may return an unrelated project's same-id session).
- L9 `sessions.rb:157-159` ‚Äî an empty-string `CLAUDE_CONFIG_DIR` is treated as set, resolving the projects dir to `/projects`.
- L10 `testing/session_store_conformance.rb:201-212` ‚Äî conformance drift: Ruby asserts 15 contracts vs Python's 14; Python's own reference store fails Ruby's contract 15 (a compatibility trap for third-party store authors).

**Public API / docs**
- L11 `lib/claude_agent_sdk.rb:250, 522` ‚Äî external iteration (`.next`/`.peek`) of the enumerators returned by `query()` is broken (fiber crosses the reactor). Docs should state only internal iteration is supported.
- L12 `lib/claude_agent_sdk.rb:57-94` ‚Äî `query()`'s YARD doc block is orphaned onto `list_sessions`; rubydoc.info renders it wrong.
- L13 `docs/observability.md` ‚Äî omits factory-observer support and the concurrency hazard of sharing one stateful OTelObserver instance (related to H5).

## 5. Test Gaps (by risk)

1. **Control-protocol round trip** (`send_control_request` ‚Üî `handle_control_response`) has zero tests ‚Äî the SDK's most central mechanism (high).
2. **`handle_permission_request`** (can_use_tool wire conversion) completely untested ‚Äî exactly why H9 survived so long (high).
3. **CLI death mid-stream**: ProcessError-before-result must unblock pending control requests / after-result must be suppressed by design ‚Äî neither direction tested (high, related to H2).
4. **`Query#close` unblocking a blocked `receive_messages` consumer** ‚Äî this hang has a documented fix history yet no regression test (high).
5. Transport read/write error paths: non-zero-exit ProcessError with stderr, 1MB buffer overflow, multi-line JSON accumulation, write guards (medium).
6. `wait_for_result_and_end_input` behavior (stdin close timing √ó hooks/SDK MCP) ‚Äî H3's regression defense (medium).
7. Hook callback raising StandardError ‚Üí error control_response path (medium).
8. SDK MCP routing error responses (unknown server/method ‚Üí -32601, handler exception ‚Üí -32603) (low).
9. Close escalation ladder (graceful ‚Üí SIGTERM ‚Üí SIGKILL) and stderr-callback isolation (low).
10. Existing test bugs: M20a (passes via race), M20b (stubs contradict reality), `subprocess_cli_transport_spec.rb:607-640` deadlock regression test has vacuous-pass risk.
